# 为什么不做"爬虫公示墙"

这是一个早期讨论中反复被提起的功能：建一个公开的网站，把已知的爬虫团体、付费群、搬运站点列出来，让大家避雷。

我们认真考虑过，最终决定**不做**。理由如下。

## 1. 公示墙比创作者更危险

被列名的团体会有强烈的反扑动机——他们在赚钱。反扑的形式包括但不限于：

- 反向举报网站本身（"传播未授权信息""诽谤"）；
- 对运营者发起人肉与骚扰；
- 在平台层面（域名、托管、CDN）发起投诉，让网站难以维持。

而创作者本人的处境往往比一个 GitHub 项目维护者更脆弱。把对抗集中到一个"公示节点"上，等于把所有人的风险集中到了那个节点上。

## 2. "匿名上传"几乎做不到真匿名

很多人会想："那让创作者匿名上传爆料就好了。"

实际上，让普通用户做到真匿名需要：Tor 浏览器 + 一次性身份 + 文本去风格化 + 设备隔离。对绝大多数创作者门槛过高。而**任何低于这个门槛的"匿名"，都给了对手反查的余地**——一旦你上传一条"我的某某作品被爬了"，这个声明本身就把作品和你的笔名绑定了，比你藏在原本的笔名后面更容易被定位。

## 3. 法律风险

未经审判把人/团体挂上标签，在中国大陆法下属于名誉侵权风险点。即使你手里证据充分，运营者也要承担举证责任，被告的可以是网站本身（以及它的实际控制人）。

## 4. 心理闭环不是只能靠"公示"实现

最初提出公示墙的核心动机是："只检测不行动，会让创作者感到无力。"

这个判断是对的。但闭环可以用别的形式提供：

- **具体的下一步**：检测到被搬运后，工具给出可执行的清单（投诉模板、平台举报入口、证据打包）。
- **手法科普**（而非主体公示）：把"这类爬取通常长什么样、付费群通常怎么运作"写成科普文档，让更多人看得懂。这部分见 [`docs/patterns/`](patterns/)。
- **互助而非示众**：未来若实现"私密预警网络"，让被同一波爬取波及的创作者之间脱敏互通信息——这是闭环，且不暴露任何人。

## 我们怎么处理仓库 issue

- `fic-guard` 的 issue 区**不接受**指名道姓任何用户 / 团体 / 群组的内容。
- 你可以描述一种**手法或模式**（"最近发现有一类付费群，做法是 XXX"），但不要带可识别的主体信息。
- 违反此规则的 issue 会被关闭并隐藏。这不是为了保护那些团体，是为了保护你和这个项目本身。

如果你确信某个具体团体在违法，请走平台举报、DMCA、或律师渠道——这些渠道有正当程序，而 GitHub issue 没有。
