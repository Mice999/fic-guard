# Threat Model

> 我们假设的对手是谁，他们能做什么，这个工具能挡住什么。

## 我们保护的对象

一名 fiction 写作者，希望：

1. 减少作品被未授权搬运的概率；
2. 一旦被搬运，能够发现，并有一份能证明"我先写的"的证据；
3. 自己的现实身份不会因为这次创作被关联出来。

## 我们假设的三种对手

### 对手 A：懒爬虫 / 文本聚合器

特征：自动化抓公开内容、做关键词匹配、批量重发到自己控制的站点或群组。不会人工审阅每篇内容，不会主动去除水印。

`fic-guard` 对这类对手**有效**：
- 零宽水印会原封不动地被搬过去，事后可提取。
- 签名句很容易被搜索引擎反查到。
- 时间戳证明你先持有。

### 对手 B：有备而来的搬运者

特征：知道有水印这种东西、会用工具批量清洗文本（替换标点、移除零宽字符、改写段落开头）。

`fic-guard` 对这类对手**部分有效**：
- 零宽水印会被清掉。
- 签名句中较长、独特性强的部分仍可能在轻度改写后被检索到。
- 时间戳证据不受影响。
- 真正的对抗手段不是技术，是法律 / 平台投诉。

### 对手 C：定向骚扰 / 去匿名化

特征：盯着特定的作者，目的不是搬运内容，而是把作者本人挖出来。可能通过截图传播、买通群组、跨平台 ID 比对实现。

`fic-guard` 对这类对手**几乎无效**——它不是一个匿名化工具。能帮上的只有 `safe-publish` 清单提醒你别在作品里留可关联现实身份的信息。如果你面对的是这类风险，技术工具远远不够；更重要的是身份隔离策略（独立邮箱、独立浏览器 profile、独立支付渠道、独立社交账号）。

## 我们明确不解决的问题

- **阻止爬取本身。** 公开可见的文本无法被技术手段彻底防爬。
- **批量起诉爬虫团体。** 这是律师和平台的事，工具能做的是帮你保留证据。
- **匿名通信。** 如果你需要匿名通信，请用 Tor + 一次性邮箱，而不是这个工具。
- **替你判断什么内容该发。** 这是你的决定，不是工具的决定。

## 我们假设的环境

- 你有一台你信任的设备来运行 `fic-guard`。
- 你的 `.fic-guard/` 目录是私密的，不会被同步到公开仓库。
- 如果你用 `fic-guard monitor --network`，你的网络出口会暴露给搜索引擎；这通常不是问题，但如果你需要更高的匿名性，请通过 Tor 跑离线模式 + 手动打开 URL。
