# Pattern: 抓取后付费打包

## 模式描述

1. 攻击方从公开站点批量抓取作品（往往是热门 CP / 热门作者的合集）。
2. 二次包装：导出为 PDF / EPUB / MOBI，整理目录，加上自己的 LOGO 或群信息。
3. 在通讯软件 / 网盘 / 二手交易平台分发，以"资源整合""合集"名义售卖或换取群费、积分。

## 识别信号

- 出现"XX CP 全集""XX 作者作品集合"等命名规范化、规模化的资源；
- 资源里同时包含多位作者、不同站点的作品（说明是批量抓取而非粉丝整理）；
- 资源附带的 LOGO / 水印是分发方自己的，而非任何原作者的；
- 分发渠道封闭（群内 / 私聊 / 付费），不在公开站点流通。

## 对创作者的影响

- 阅读量下降（读者绕过原站点）；
- 你失去对作品的删改控制（即使你在原站点改了/删了，他们手里还有旧版本）；
- 你的内容可能出现在你不希望出现的语境里（被收录进与原意完全不符的合集）。

## 应对

**预防层**：

- 把作品设为"仅注册用户可见"（提高抓取门槛）；
- 用 `fic-guard fingerprint watermark` 给不同平台的发布版本嵌入不同标签——如果合集里能提取出某个特定标签，就知道泄漏源；
- 用 `fic-guard timestamp make` 留好原创证据。

**事后层**：

- 用 `fic-guard fingerprint extract` 在拿到的合集样本上提取水印；
- 用 `fic-guard monitor` 定期把签名句往搜索引擎扔，能抓到被传播到半公开渠道的部分；
- 按 [`incident-response.md`](../incident-response.md) 走投诉流程。

## 局限

封闭渠道里的分发，工具基本看不到。这部分要靠社区自发观察。`fic-guard` 不会主动潜入任何此类渠道。
