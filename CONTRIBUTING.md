# Contributing to fic-guard

欢迎贡献！但有几条规则我们坚持，请在 PR / issue 前阅读。

## 我们接受的贡献

- bug 修复、文档改进、测试补充；
- 新功能 PR（先开 issue 讨论再写代码，避免做完了发现方向不对）；
- 新增 `docs/patterns/` 下的手法/模式描述（**严格遵守该目录 README 的规则**）；
- 翻译（特别是英文 README，以便圈外读者也能理解我们在做什么）。

## 我们不接受的贡献

以下类型的 issue / PR 会被关闭，不解释：

1. **指名道姓任何用户、团体、群组、域名**。无论你的意图多么正当，这个仓库不是举报平台。如果你想曝光，请走平台投诉或法律渠道。
2. **要求加入"公示墙"功能**。我们已经在 [`docs/why-not-a-callout-site.md`](docs/why-not-a-callout-site.md) 中详细说明了不做的原因。
3. **要求加入主动爬取/对抗第三方站点的功能**。`fic-guard` 是作者侧的自保工具，不做对抗性的网络行为。
4. **包含具体侵权内容的截图**（除非已经做了充分脱敏，且只是为了说明模式）。
5. **政治化的"立场宣言"PR**。我们刻意保持工具中性。

## 开发流程

```bash
git clone https://github.com/<owner>/fic-guard.git
cd fic-guard
pip install -e ".[dev]"
pytest tests/
```

新代码需要：

- 通过现有测试；
- 为新功能补充测试；
- 不引入新的网络依赖（除非通过 `--network` 这样的明确开关 opt-in）；
- 不引入会本地写入任何"用户身份"信息的代码。

## 报告安全问题

如果你发现了一个真正的安全 / 隐私漏洞（例如：某条命令会意外把用户的本地文件路径泄露到网络请求里），**不要开公开 issue**，请用 GitHub 的 Security Advisory 私下报告。
